## What you are going to build

By the end of this guide, two processes will be running inside your ICRN JupyterLab session:

- **Terminal 1**: `vllm serve` — a self-hosted OpenAI-compatible LLM server on port `8000`. This is the "self-hosted LLM serving" your project is really about.
- **Terminal 2**: `uvicorn app.main:app` — your FastAPI backend on port `8001`. It talks to the vLLM server on `localhost:8000` and exposes `/api/chat` to you.

You will `curl` your backend from a third terminal and see an answer generated by **Qwen2.5-7B-Instruct running on the H200 GPU** — no OpenAI, no mock, real self-hosted inference.

## Step 4 — Launch vLLM in Terminal 1

Still in `~/IlliniGuide/backend/` with the venv active:

```bash
vllm serve Qwen/Qwen2.5-7B-Instruct \
    --host 0.0.0.0 \
    --port 8000 \
    --dtype float16 \
    --max-model-len 8192 \
    --gpu-memory-utilization 0.85 \
    --enable-prefix-caching
```

**What each flag means** (this is the vLLM interview material):

| Flag | Value | Why |
|---|---|---|
| `Qwen/Qwen2.5-7B-Instruct` | model id | HuggingFace repo. vLLM will download to `~/.cache/huggingface/` on first launch (~15 GB, ~5–10 min). Subsequent launches reuse the cache. |
| `--host 0.0.0.0` | listen on all interfaces | Otherwise vLLM only binds to `127.0.0.1`. Since both processes are in the same container, `127.0.0.1` would technically work too, but `0.0.0.0` is the more general "let anyone reach me on this machine" choice. |
| `--port 8000` | HTTP port | The `/v1/chat/completions` endpoint will live at `http://localhost:8000`. Our backend's `VLLM_BASE_URL` will point here. |
| `--dtype float16` | fp16 weights | 2 bytes per param → 14 GB weights for a 7B model. bf16 works too on H200; fp32 doubles memory and is wasteful for inference. |
| `--max-model-len 8192` | max sequence (prompt + generation) tokens | Bounds KV cache per request. Higher = each request may take more KV pages. 8K is plenty for advising queries. |
| `--gpu-memory-utilization 0.85` | fraction of VRAM vLLM may claim | 141 GB × 0.85 = ~120 GB. Leaves ~20 GB for other processes (safe on shared machines). vLLM plans its KV cache pool from this budget — the higher, the more concurrent requests fit. |
| `--enable-prefix-caching` | cache prefill of shared prefixes | Since every advising request shares the same system prompt, prefill for that prefix is computed **once** and reused. Big TTFT win for our workload. |

The first launch prints a lot. Look for two milestones:

1. `Downloading` messages (from HuggingFace) — this is the ~15 GB fetch. First time only.
2. Near the bottom, after a minute of PyTorch compiling kernels:

```
INFO:     Started server process [...]
INFO:     Uvicorn running on http://0.0.0.0:8000 (Press CTRL+C to quit)
```

**You should see** that final `Uvicorn running on` line. Don't touch this terminal after that — vLLM is now serving.

## Step 6 — Peek at vLLM metrics (teaching)

vLLM exposes Prometheus-format metrics at `/metrics`. This is where all the concepts you learned (KV cache usage, TTFT, throughput) show up as numbers.

```bash
python -m scripts.vllm_metrics_snapshot
```

**You should see** a small report like:

```
=== vLLM metrics snapshot @ http://localhost:8000/metrics ===
model:                       Qwen/Qwen2.5-7B-Instruct
requests running:            0
requests waiting:            0
GPU KV cache usage:          0.0%     ← empty because no requests are running
prefill tokens (total):      12
decode tokens (total):       33
prompt throughput (tok/s):   ...
generation throughput (tok/s): ...
avg time to first token:     ...
```

**What to notice** (this is you validating Task C0):

- `GPU KV cache usage` % is the direct empirical measurement of the KV-cache-fills-up-first bottleneck we discussed. Right now it's near 0 (nothing running). Under 10-concurrency load (Task C6) you will see it climb to 60-90%.
- `prefill tokens (total)` vs `decode tokens (total)` — the two phases you learned about. Under real load `decode` will vastly outgrow `prefill` because each request generates many more tokens than its prompt.
- `avg time to first token` is TTFT. Under prefix caching (which we enabled) it should be very small for repeat system prompts.

Run the snapshot again after Step 8 to see the numbers move.

## Troubleshooting

### "No CUDA devices found" / `torch.cuda.is_available() == False`

Your JupyterLab session did not get a real GPU. Either:

- You picked the CPU-only environment by accident → go back to Step 0 and pick H200.
- The H200 pool was full → the environment might have silently given you CPU. Confirm with `nvidia-smi`.

### `vllm serve` says `OSError: We couldn't connect to https://huggingface.co ...`

Rare on ICRN but possible if outbound HuggingFace is blocked. Try downloading the model with a token first:

```bash
huggingface-cli login       # paste an HF read token
huggingface-cli download Qwen/Qwen2.5-7B-Instruct
```

Then re-run `vllm serve`. If HF is fully blocked, ping me and we'll pick a mirror.

### `CUDA out of memory`

Two knobs to turn down (in order):

1. `--gpu-memory-utilization 0.7` (from 0.85) — leaves more headroom, useful on shared nodes.
2. `--max-model-len 4096` (from 8192) — smaller max KV cache per request.

If OOM persists on a 141 GB H200 with a 7B model, something else on the node is hogging VRAM. Check `nvidia-smi` for other processes.

### `address already in use` on port 8000

Somebody else is using that port. Change to `--port 8100` in Step 4 and update `VLLM_BASE_URL` in Step 7 accordingly.

### First request from backend takes ~30 seconds

Expected on the very first call — PyTorch is still JIT-compiling some kernels for the actual sequence length. Second call onward should be a couple of seconds for a normal advising answer.

### Backend response still says `[mock backend=...]`

You forgot to `export LLM_BACKEND=vllm_remote` in Terminal 3 before running uvicorn, or you have a stale process from before. `pkill -f uvicorn` and restart.

### vLLM server logs errors like `Engine iteration timed out`

The model is stuck on a request (maybe max_tokens too high on a slow prompt). Restart Step 4. If it happens often, drop `--max-model-len` to 4096 and `--gpu-memory-utilization` to 0.8.

### `graceful degradation` message: LLM call failed, template answer returned

Your backend could reach vLLM, but vLLM returned an error. Check Terminal 1 (vLLM logs) for the actual exception. Common causes: model_name mismatch (backend sent `Qwen/...` but vLLM was launched with a different id), or 4xx from a malformed message.
